## はじめに

Claude Code のような AI コーディングエージェントに開発を任せるようになって、あることに気づきました。

**AI がいちばん時間を溶かすのは、コードを書くことではなく「コマンドが動かない」ことへの対処です。**

- `python` を叩いたらホストに Python が入っていない
- venv を activate し忘れて `ModuleNotFoundError`
- ネイティブライブラリが無くて `OSError: cannot load library 'libgobject-2.0-0'`
- テストの実行方法がリポジトリごとに違い、README を掘って推測で叩く

これらは人間の新規メンバーがオンボーディングでハマるポイントと完全に同じです。つまり「AI に優しい環境」と「人間に優しい環境」は同じ問題を解けばよく、私は **flake.nix + Makefile** の 2 枚で解くのが今のところ一番手堅いと思っています。

この記事では、個人開発中のプロジェクト（FastAPI + React + OpenTofu 構成、以下 DevForge）で実際に運用している構成を紹介します。

## 全体の設計

役割分担はシンプルです。

| ファイル | 責務 |
|---|---|
| `flake.nix` | **何が入っているか**の正本。ツールチェーン（Python / Node / CLI / ネイティブライブラリ）をバージョン込みで宣言する |
| `Makefile` | **どう実行するか**の正本。すべてのターゲットを `nix develop --command` でラップする |
| `CLAUDE.md` | AI への入口。「コマンドは `make` 経由で実行せよ」と書くだけ |

ポイントは、**ホスト OS に開発ツールを一切インストールしない前提**に倒すことです。Python も Node も ruff も OpenTofu も、すべて Nix devshell が提供します。人間も AI も「リポジトリを clone して `nix develop`」だけで同じ環境に立てます。

## flake.nix — ツールチェーンの正本

DevForge の `flake.nix` の devShell 部分を簡略化するとこうなります。

```nix
{
  description = "DevForge 開発環境";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs = { self, nixpkgs, flake-utils }:
    flake-utils.lib.eachDefaultSystem (system:
      let
        pkgs = nixpkgs.legacyPackages.${system};

        # WeasyPrint（PDF 生成）が必要とするネイティブライブラリ
        weasyPrintLibs = with pkgs; [
          pango cairo glib gobject-introspection
          libffi gdk-pixbuf fontconfig freetype harfbuzz
        ];
      in
      {
        devShells.default = pkgs.mkShell {
          packages = with pkgs; [
            # --- 言語ランタイム ---
            python313
            nodejs_22          # Node.js 22 LTS（npm 同梱）

            # --- ミドルウェア / CLI ---
            redis
            turso-cli
            stripe-cli
            opentofu           # Terraform 互換の IaC ツール

            # --- 共通ツール ---
            git
            gh
            gnumake
          ] ++ weasyPrintLibs;

          # ネイティブライブラリを動的リンカーに見せる
          # （macOS の dyld は LD_LIBRARY_PATH を無視するため DYLD_* も設定）
          shellHook = ''
            export LD_LIBRARY_PATH="${pkgs.lib.makeLibraryPath weasyPrintLibs}:$LD_LIBRARY_PATH"
            export DYLD_FALLBACK_LIBRARY_PATH="${pkgs.lib.makeLibraryPath weasyPrintLibs}:''${DYLD_FALLBACK_LIBRARY_PATH:-}"

            echo "DevForge 開発環境"
            echo "  Python : $(python3 --version)"
            echo "  Node   : $(node --version)"
            echo "セットアップ: make setup"
          '';
        };
      }
    );
}
```

### なぜ Docker ではなく Nix なのか

「環境の再現なら Docker でよくない？」とよく言われますが、開発シェルの用途では Nix に分があります。

- **エディタや AI エージェントがホスト側で動く**。コンテナ内開発は VS Code の Remote Container や AI ツールとの接続に一手間かかるが、devshell はただのシェル環境なのでそのまま繋がる
- **ファイル監視・Git・ネットワークがネイティブ**。macOS の Docker のファイル I/O の遅さと無縁
- **`flake.lock` で nixpkgs のリビジョンごと固定される**。「同じ Dockerfile なのに build する日によって結果が違う」が起きない

DevForge では Docker も使いますが、役割は「本番同等のランタイム検証（compose での API + Redis + libSQL 起動）」に限定し、開発ツールチェーンは Nix に寄せています。

### direnv で入場を自動化する

リポジトリ直下に 1 行だけの `.envrc` を置いておくと、`cd` するだけで devshell に入れます。

```bash
# .envrc
use flake
```

初回に `direnv allow` するだけ。人間は「devshell に入り忘れた」というミスから解放され、VS Code も direnv 拡張経由で正しい Python を解決できるようになります。実際 DevForge では、`.vscode/settings.json` に `.venv/bin/python` の絶対パスがハードコードされていた問題を、これで根本から消しました。

## Makefile — 実行方法の正本

flake.nix だけだと「devshell に入ってからコマンドを叩く」という手順が残ります。ここを埋めるのが Makefile で、**全ターゲットを `nix develop --command` でラップ**します。

```makefile
# CI 相当を一括実行
ci: lint test build-web

test-backend:
	nix develop --command bash -c "cd backend && python -m pytest -q --cov=app tests"

test-web:
	nix develop --command bash -c "cd web && npm test"

lint-backend:
	nix develop --command bash -c "cd backend && ruff check app tests"

migrate:
	nix develop --command bash -c "cd backend && alembic upgrade head"

infra-validate-dev:
	nix develop --command bash -c "tofu -chdir=infra/environments/dev init -backend=false && tofu -chdir=infra/environments/dev validate"
```

こうしておくと、**devshell の外から `make test-backend` と叩いても正しく動きます**。「シェルの状態」という暗黙の前提が消えて、コマンドが自己完結するのがミソです。

もうひとつ、デフォルトターゲットは `help` にして、ターゲット一覧を日本語で出すようにしています。

```makefile
help:
	@echo "テスト・リント"
	@echo "  ci                lint + test + build-web を一括実行 (CI 相当)"
	@echo "  test-backend      Backend: pytest"
	@echo "  test-web          Frontend: vitest"
	@echo "  lint-fix          リント自動修正 (ruff + eslint)"
	...
```

`make` と打てば「このリポジトリでできること」の全カタログが出る。人間にとってはオンボーディングドキュメントであり、AI にとっては**利用可能なツールの一覧**として機能します。

## AI にはこう効く

CLAUDE.md（AI エージェント向けのプロジェクト指示書）には、実質これだけ書けば済むようになります。

> **原則: 開発ツールはすべて Nix devshell 経由で実行する。** ホスト側に Python / Node / ruff / tofu は入っていない前提。
>
> 第一選択は `make` ターゲット（`nix develop --command` でラップ済み）。make に無い操作のみ `nix develop --command bash -c "..."` を使う。生シェルでの直接実行は禁止。

効果は体感でハッキリ出ます。

**1. コマンドの語彙が固定される**

AI は「テストどう走らせるんだっけ」を推測しなくなります。`make ci` / `make test-backend` / `make lint-fix` という決まった語彙があるので、探索のための無駄なツール呼び出し（= トークンと時間）が減ります。

**2. 「動かない」の再現と説明が一意になる**

環境がロックされているので、AI の手元で落ちたテストは私の手元でも同じように落ちます。「たぶん環境の問題です」という一番不毛な会話が消えます。

**3. 失敗パターンを 1 箇所に書ける**

たとえば DevForge では、nix の外で pytest を叩くと WeasyPrint が `OSError: cannot load library 'libgobject-2.0-0'` で落ちます。この失敗はエラーメッセージごと CLAUDE.md に書いてあり、AI は踏む前に回避します。環境が発散していたら、こういう「地雷マップ」はそもそも書けません。

**4. `make ci` が完了報告のゲートになる**

「改修後は `make ci` を pass させてから完了報告する」というルールにしています。CI 相当がローカルの 1 コマンドに集約されているから成立するルールで、push してから CI が落ちて手戻り、というループが大幅に減りました。

## 人間にはこう効く

そして同じ仕組みが、そのまま人間のためになります。

- **オンボーディングが 3 コマンド**: `nix develop` → `make setup` → `make dev`。README の「インストール手順」章はほぼ消滅
- **マシン乗り換え・OS 再インストールが怖くない**: 環境は全部リポジトリの中にある
- **`make help` が生きたドキュメント**: コマンドと説明が同じファイルにあるので、ドキュメントだけ古くなる事故が起きにくい
- **チートシート不要**: プロジェクトごとに `pytest` の起動オプションや lint の流儀を覚えなくていい。全部 `make` の後ろに隠れている

## 発展: Python の依存パッケージまで Nix に寄せる

ここまでは「ツールチェーンを Nix で配る」話でしたが、DevForge ではさらに一歩進めて、**Python の依存パッケージ本体も [uv2nix](https://github.com/pyproject-nix/uv2nix) で Nix build し、`.venv` を廃止**しました。

```nix
# pyproject.toml + uv.lock を読んで、全依存入りの virtualenv を Nix で構成
workspace = uv2nix.lib.workspace.loadWorkspace { workspaceRoot = ./backend; };
pythonSet = (pkgs.callPackage pyproject-nix.build.packages {
  python = pkgs.python313;
}).overrideScope (lib.composeManyExtensions [
  pyproject-build-systems.overlays.default
  (workspace.mkPyprojectOverlay { sourcePreference = "wheel"; })
]);
backendEnv = pythonSet.mkVirtualEnv "devforge-backend-env" workspace.deps.default;
```

この `backendEnv` を devShell の `packages` に入れると、`pip install` も `uv sync` も一切なしで、devshell に入った瞬間から pytest / ruff / alembic が依存込みで使えます。依存の正本は `pyproject.toml` + `uv.lock` のままなので、普段の依存追加は uv の流儀のまま、環境構築だけが Nix に置き換わる形です。

さらに同じ flake から本番用のランタイム環境（`nix build .#backend-runtime`）も出力していて、**開発・CI・本番イメージが同一のロックファイルから構成される**ところまで持っていけました。「開発では動くのに本番イメージだけ segfault する」という類の事故（実際に Python バージョン bump で一度踏みました）を、構造ごと潰せます。

このあたりは影響範囲が大きいので、Phase 0〜3 の段階移行として ADR（Architecture Decision Record）を書いてから進めました。いきなりやるものではないですが、「flake.nix + Makefile」の土台があると、この手の移行も差分が局所化されて進めやすくなります。

## ハマりどころ

いいことばかり書いたので、実際に踏んだ注意点も。

- **Nix の学習コストは正直高い**。ただし devShell + `packages` のリストだけなら Nix 言語の深淵に触れずに書けます。まずそこだけ真似するのがおすすめ
- **初回の `nix develop` は遅い**。依存のダウンロードとビルドが走るため数分かかることがあります。2 回目以降はキャッシュで一瞬
- **ネイティブライブラリは `shellHook` での環境変数設定が要る**ことがある（WeasyPrint の例のように）。エラーメッセージごと CLAUDE.md に記録しておくと、未来の自分と AI が救われます
- **サンドボックス系ツールとの相性**。Claude Code のサンドボックスが nix のロックファイル書き込みをブロックするケースがあり、これも CLAUDE.md に対処法を明記して運用しています

## まとめ

- **flake.nix** に「何が入っているか」、**Makefile** に「どう実行するか」を寄せると、環境の暗黙知がリポジトリ内のテキストになる
- 暗黙知がテキストになると、人間のオンボーディングが縮み、AI エージェントの探索コストと事故が減る。**両者は同じ問題**
- AI への指示は「`make` 経由で実行、生シェル禁止」の数行に集約できる
- 発展形として、uv2nix で言語の依存パッケージや本番イメージまで同じ flake に寄せる道もある

AI にコードを書かせる時代の開発環境整備は、「AI 専用の何か」を用意することではなく、**昔ながらの再現性・自己文書化を極端なまでに徹底すること**なんだな、というのが運用してみての実感です。
