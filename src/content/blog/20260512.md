結論から言うとこの案件は途中でお断りされる形になった。

githubにpushすることができたから確認して欲しいと連絡があり、
コードを開いた瞬間から、想定とは全然違う景色が広がっていた。

---

## 最初に気づいたこと

リポジトリにファイルが1つしかない。`App.js`(例) だ。

行数を確認した。**6,000行超**だった。

TypeScriptは使われていない。`.js` のまま。おそらくUIテンプレートを最初に生成したものが肥大化したのだろうと思う。
スタイル定義が500行以上同じファイルに混在している。同名のスタイルキーが複数箇所で重複定義されている。ほぼ同じ処理の画面コンポーネントがコピペで2つある。

バックエンドも見た。こちらもアプリケーションから呼ばれているpyモジュールが1つ`api.py`(例) 約1,500行。

こちらは意外にも、基本的なセキュリティ対策がある程度実装されていた。

- DB設定は環境変数から取得されている
- パスワードはPBKDF2でハッシュ化されている
- タイミング攻撃対策として`hmac.compare_digest`が使われている
- Pydanticで型定義されている

ただし、運用上の問題は別の形で現れていた。
```python
# テスト用の固定IDが本番コードに残ったまま
FALLBACK_USER_ID = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

# autocommitがTrueで全操作が即座にコミットされる
conn.autocommit = True

# エラーの詳細を握りつぶす空のexceptが複数箇所
except Exception:
    return {}
```

ここで気づいたのは、FEとBEで技術負債の性質が非対称だということだ。BEはセキュリティの基礎がある程度できている一方、FEは6,000行モノリスで設計という概念がない。この非対称性が、改修の難しさをさらに複雑にしていた。

これは怠慢の結果ではない。**AIに設計を委ねたまま追記し続けた結果**でリファクタリングという概念がないのだろう。
AIは動くことを最優先するが肥大化したコードは自動的にリファクタリングしてくれない。

---

## AI生成コードの特徴的なパターン

実際に目にしたパターンを整理する。

**コメントがない**

AIはコードを生成するとき「なぜそう書くか」を持っていない。動くコードを出力するだけだ。結果として、何百行にもわたるコードに意図の説明が一切ない。

**同名の変数・関数が乱立する**

AIはリクエストのたびに「とりあえず動くコード」を生成する。前後の文脈を持たないので、同じ名前の変数が複数の関数に独立して定義される。

```js
// 画面Aのコンポーネント
function ScreenA() {
  const [data, setData] = useState(null)
  // ...
}

// 画面Bのコンポーネント（別の箇所で同じ変数名）
function ScreenB() {
  const [data, setData] = useState(null)
  // ...
}

// 画面Cのコンポーネント（さらに同じ変数名）
function ScreenC() {
  const [data, setData] = useState(null)
  // ...
}
```

**コピペが横展開される**

ほぼ同じ処理の画面が2つある。共通化という発想がない。AIは求められた機能を都度生成するだけで、重複を整理するタイミングがない。

---

## なぜこれが問題か

「動いている」は事実だ。アプリとして起動するし、APIも応答する。

ただ「運用できる」とは全然違う話。

- バグが出たとき、6,000行のどこで何が起きているか追えない
- 1行直すと別の何かが壊れる。影響範囲がわからない
- テストが書けない構造になっている
- BEのセキュリティ基礎はあるが、運用上の問題（グローバル接続使い回し、本番直結、テスト用IDの混入）が把握されていない

開発環境すらない。ローカルで動かすたびにグローバルの環境が汚染されていく。

---

## 「動いているから軽微な改修」という誤解

今回の依頼は「課金機能の追加」だった。既存のコードベースに機能を追加するだけ、という前提がクライアント側にあった。

実態は違う。

6,000行の型なしコードに課金処理を追加するのは、新規プロジェクトで実装するより遥かにコストが高い。影響範囲が把握できないからだ。どこかに変数が衝突しているかもしれない。状態管理がどこで何をしているか追うだけで時間がかかる。

「動いているコード」の改修コストを正確に見積もるには、まずコードを読む時間が必要だ。それ自体がコストだということが、伝わらないことが多い。

---

## テストが1行もなかった

コードを一通り読んで、最後に確認したのがテストの有無だった。

`tests/` ディレクトリはなかった。テストファイルも1行も存在しなかった。

これは書かなかったのではなく、**書けない構造になっている**からだ。

```python
# 本来あるべき構造
# routers/auth.py   → エンドポイント定義のみ
# services/auth.py  → パスワード検証などのビジネスロジック
# repositories/user_repo.py → DBアクセスのみ

# 実態：1関数にすべてが混在
@app.post("/auth/login")
def login(body: LoginRequest):
    # DB接続（Repository層）
    conn = get_db_connection()
    cur = conn.cursor()
    cur.execute("SELECT user_id, password_hash FROM users WHERE email = %s", (body.email,))

    # ビジネスロジック（Service層）
    row = cur.fetchone()
    if not verify_password(body.password, row[1]):
        return {"error": "invalid_credentials"}

    # トークン生成（Service層）
    token = generate_token(user_id=row[0])
    return {"access_token": token}
```

API層・Repository層・ビジネスロジックが1関数に混在しているので、ログイン処理を単体でテストしようとすると必ず本番DBが必要になる。依存関係を差し替える設計になっていないからだ。

AIは「テストを書いて」と明示しない限りテストを生成しない。「動くコードを作って」というプロンプトにテストは含まれていない。結果として、動く → リリース → 次の機能追加というサイクルだけが回り続ける。

動いているように見えるプロダクトが、一度も検証されていないコードの上に立っていた。

## どうすればいいのか

まず、前提として個人開発の文脈で、
この状況を改善するには、一気にリファクタリングをするのではなくモノリスでプロトタイプを作り切った後リファクタするサイクルを回すことが必要だと思った。

**この案件から学んだAIコード技術負債を減らすための開発初期でやるべきポイント**

**プロトタイプができたら、機能単位でコードを分割する**

例えば、コアになる機能userフォーム入力しDBにAPIを叩いて保存するのような機能があったとする。
最小の段階はUI1ファイル、API層→repository層1ファイルで構成していても良いが、動くものができた時点で
フレームワークのベストプラクティスに沿って、**機能単位でコードを分割する**ことが重要だと思う。
ここは今までのシステム開発手法（**設計⇨構築**）とは異なる部分で個人開発でプロトタイプが民主化されたこの時代には**構築→設計**のサイクルを回すことが重要だと思う。
ここでの設計方針などは.claude/rulesなどでプロンプトに入れておくと、リファレンスとして機能するのかなと思う。（筆者はclaude userなので）

**新しい機能は分離して書く**

次にこれから追加する機能は前段で設計したベースをもとに機能追加してもらうように機能ごとに
api / service / repository などを分けて、テストが書ける形にする。既存コードには触れない。
正直このフェーズは共通的に書くというのは悠然度を下げて動くものができたら、前段の**構築→設計**に立ち返る

**Gitを使いましょう**

開発初期こで一人で開発する場合でも、GitHubなどのバージョン管理ツールを使うことは非常に重要だと思う。
コードの中にコンテキストがなくてもgitのコミットログやブランチ構成から、どのような機能がいつ追加されたのか、どのような変更があったのかを把握することができる。

## おわりに

非エンジニアがサービスをリリースすること**プロトタイプの民主化**により今回紹介した事例は極端だが多く存在していると思う。

ポジションのような締めくくりにはしたくないし、そんなポジションが自分にあるとは思えないが
AIで生成されたサービスもおおよそ従来のwebサービスの形態を踏襲しているため
今まで通りのシステム設計のベストプラクティスを適用することが、技術負債を減らすためには重要だと思う。

ぜひAIコーディングでその学習きっかけになりますように。
